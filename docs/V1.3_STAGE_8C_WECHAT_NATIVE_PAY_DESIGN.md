# V1.3 Stage 8C 微信支付 Native 二维码方案设计

## 1. 背景

V1.3 Goal Fit 已经完成启动页、测试页、免费结果页、解锁确认页和完整报告页的前端闭环。当前 Stage 8B 的 Unlock Flow Shell 只是为了把用户路径从“免费结果页直接解锁”改成“进入解锁确认页再解锁”，方便后续接入真实支付。

当前链路是：

```text
启动页
↓
测试页
↓
免费结果页
↓
解锁确认页
↓
前端确认解锁
↓
完整报告
```

Stage 8B 中的解锁仍然是前端本地状态，不是支付系统。本设计文档用于后续 Stage 8D / 8E / 8F / 8G 接入真实微信支付前统一产品、工程、安全和数据方案。

## 2. 当前状态

当前免费结果页点击“解锁完整目标适配报告 ¥19.9”后，会进入解锁确认页。用户点击确认后，前端会写入本地订单占位状态和本地 unlock 标记，然后进入完整报告。

当前实现的边界：

- 没有调用微信支付接口。
- 没有真实订单后端。
- 没有数据库。
- 没有支付回调。
- 没有服务端 entitlement。
- `localStorage` unlock 只用于前端模拟解锁。

真实支付上线后，完整报告权限不能继续依赖前端 `localStorage`。`localStorage` 可以作为前端体验缓存，但不能作为最终解锁凭证。

## 3. 核心决策

### 3.1 第一版真实支付采用微信支付 Native 二维码模式

第一版真实支付优先采用微信支付 Native 支付二维码模式。

原因：

- 用户可以在浏览器中看到二维码，再使用微信扫码完成支付。
- 不强依赖公众号网页授权。
- 不需要第一版就接入 openid 链路。
- 适合当前 Vercel H5 页面 + 解锁确认页的产品形态。

### 3.2 第一版暂不采用 JSAPI 支付

JSAPI 更适合微信内网页，但需要公众号网页授权和 openid 链路。第一版如果直接接 JSAPI，需要处理授权、openid、支付目录、用户环境判断等额外链路，复杂度更高。

因此第一版先不采用 JSAPI。后续如果产品确认主要在微信内传播，并且需要更顺滑的微信内支付体验，可以在 Native 跑通后再评估 JSAPI。

### 3.3 不采用固定微信收款码

固定微信收款码不适合作为本产品正式支付方式。

原因：

- 固定收款码无法自动识别是哪一个用户付款。
- 固定收款码无法自动识别是哪一个 `sessionId`。
- 固定收款码无法自动绑定是哪一份完整报告。
- 固定收款码无法稳定触发系统自动解锁。
- 固定收款码会导致人工核对、人工解锁和售后成本上升。

本产品需要“支付成功后自动解锁对应完整报告”，所以必须使用可绑定订单号和回调的支付模式。

### 3.4 不采用纯前端支付判断

纯前端支付判断不可作为真实支付方案。

原因：

- 微信支付商户密钥、API v3 密钥、商户私钥不能放在前端。
- 前端不能直接调用微信支付下单接口。
- 前端 `localStorage` 可以被清理，也可以被篡改。
- 订单状态不能只存在浏览器里。
- 完整报告权限必须由服务端确认。

### 3.5 Stage 8B 的 localStorage unlock 只是模拟解锁

Stage 8B 当前完成的是支付前端壳子，不是支付系统。真实支付上线后，unlock 权限必须从 `localStorage` 迁移到服务端 entitlement 判断。

## 4. 目标支付链路

目标链路如下：

```text
启动页
↓
测试页
↓
免费结果页
↓
解锁确认页
↓
点击确认解锁
↓
后端创建订单
↓
后端调用微信支付 Native 下单
↓
返回二维码 code_url
↓
前端展示二维码
↓
用户用微信扫码支付
↓
微信支付回调后端 notify_url
↓
后端验签、解密、确认支付成功
↓
后端更新订单 paid
↓
后端创建完整报告 entitlement
↓
前端轮询订单状态
↓
显示完整报告已解锁
↓
进入完整报告
```

## 5. 系统架构

### 5.1 前端

前端继续使用：

- Vite
- React
- TypeScript
- Vercel 部署

前端负责：

- 启动页
- 测试页
- 免费结果页
- 解锁确认页
- 二维码展示
- 订单状态轮询
- 完整报告展示

前端不负责：

- 生成真实支付订单
- 保存支付密钥
- 判断最终支付成功
- 创建最终解锁权限

### 5.2 后端

第一版建议使用 Vercel Serverless Functions。

后端负责：

- 创建订单
- 调用微信支付 Native 下单
- 接收支付回调
- 验签和解密
- 维护订单状态
- 维护完整报告解锁权限
- 对完整报告访问做服务端校验

### 5.3 数据库

建议使用：

- Supabase Postgres
- Neon Postgres
- 或其他云数据库

至少需要三类核心数据：

1. `goal_fit_sessions`
2. `goal_fit_orders`
3. `goal_fit_entitlements`

同时建议增加支付回调审计表：

4. `goal_fit_payment_notifications`

如果当前 session 仍主要存在 `localStorage`，真实支付阶段至少需要把 `sessionId`、目标公司、目标岗位、必要结果摘要保存到服务端。否则支付成功后，服务端无法稳定判断对应哪份报告，也无法在用户关闭页面后恢复解锁状态。

## 6. 数据表设计

### 6.1 goal_fit_sessions

用途：将测试结果从纯前端状态升级为服务端可识别状态。

字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 数据库主键 |
| session_id | 前端测试会话 ID，唯一 |
| target_company_code | 公司类型 code |
| target_company_label | 公司类型展示文案 |
| target_role_code | 岗位方向 code |
| target_role_label | 岗位方向展示文案 |
| answers_json | 用户答题记录，可选但推荐 |
| result_summary_json | 结果摘要，用于恢复和校验 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

如果第一版不保存完整 answers，也至少保存：

- `session_id`
- `target_company_code`
- `target_company_label`
- `target_role_code`
- `target_role_label`
- `result_summary_json`

### 6.2 goal_fit_orders

用途：记录订单和支付事实。

字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 数据库主键 |
| order_id | 系统内部订单 ID |
| out_trade_no | 微信支付商户订单号，唯一 |
| session_id | 关联测试 session |
| product_code | 商品 code |
| product_name | 商品名称 |
| amount | 金额，单位分 |
| currency | 币种 |
| status | 订单状态 |
| wechat_code_url | Native 支付二维码 code_url |
| wechat_transaction_id | 微信支付交易号 |
| created_at | 创建时间 |
| expires_at | 订单过期时间 |
| paid_at | 支付完成时间 |
| closed_at | 关闭时间 |
| last_error | 最近一次错误信息 |

`status` 枚举：

```text
pending
paid
failed
cancelled
expired
closed
```

固定商品信息：

```text
amount = 1990
currency = CNY
product_code = goal_fit_full_report
product_name = 完整目标适配报告
```

### 6.3 goal_fit_entitlements

用途：记录完整报告访问权限。

字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 数据库主键 |
| session_id | 关联测试 session |
| order_id | 关联订单 |
| product_code | 权限对应商品 |
| status | 权限状态 |
| created_at | 创建时间 |
| revoked_at | 撤销时间 |

`status` 枚举：

```text
active
revoked
```

说明：

- entitlement 表才是真正决定用户是否可以查看完整报告的依据。
- 不要用订单 `paid` 直接替代 entitlement。
- 订单代表支付事实。
- entitlement 代表访问权限。

### 6.4 goal_fit_payment_notifications

用途：记录微信支付回调，方便排查重复通知、验签失败、金额不一致、回调异常等问题。

字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 数据库主键 |
| event_id | 回调事件 ID |
| out_trade_no | 商户订单号 |
| transaction_id | 微信支付交易号 |
| event_type | 回调事件类型 |
| raw_body_json | 回调原始内容 |
| processed_status | 处理状态 |
| processed_at | 处理时间 |
| created_at | 创建时间 |

## 7. API 设计

### 7.1 创建订单

```http
POST /api/goal-fit/orders/create
```

请求：

```json
{
  "sessionId": "xxx",
  "productCode": "goal_fit_full_report"
}
```

后端逻辑：

1. 校验 `sessionId` 是否存在。
2. 校验 `productCode` 是否为 `goal_fit_full_report`。
3. 金额在服务端写死为 `1990`，不信任前端传来的金额。
4. 如果该 session 已有 active entitlement，直接返回 paid / unlocked。
5. 如果已有未过期 pending 订单，直接返回原订单和二维码。
6. 如果没有可复用订单，创建新订单。
7. 调用微信支付 Native 下单。
8. 保存 `wechat_code_url`。
9. 返回订单信息。

响应：

```json
{
  "orderId": "xxx",
  "outTradeNo": "xxx",
  "status": "pending",
  "amount": 1990,
  "currency": "CNY",
  "codeUrl": "weixin://...",
  "expiresAt": "2026-07-05T12:00:00.000Z"
}
```

### 7.2 查询订单状态

```http
GET /api/goal-fit/orders/status?orderId=xxx
```

响应：

```json
{
  "orderId": "xxx",
  "status": "pending",
  "unlocked": false,
  "reportUrl": "/result-goal-fit-preview?session=xxx"
}
```

`status` 可为：

```text
pending
paid
expired
failed
cancelled
```

前端用途：

- 二维码页面每 2-3 秒轮询一次。
- 最多轮询到订单过期。
- 如果 `status=paid` 且 `unlocked=true`，展示“完整报告已解锁”。

### 7.3 微信支付回调

```http
POST /api/wechat/pay/notify
```

后端逻辑：

1. 读取微信支付通知。
2. 验证签名。
3. 解密 resource。
4. 根据 `out_trade_no` 找到订单。
5. 校验金额是否为 `1990`。
6. 校验 `product_code` 是否为 `goal_fit_full_report`。
7. 幂等处理：如果订单已经 paid，直接返回成功。
8. 更新订单为 paid。
9. 写入 `wechat_transaction_id`。
10. 创建或确认 entitlement active。
11. 记录 payment notification。
12. 返回微信支付要求的成功响应。

### 7.4 检查报告权限

```http
GET /api/goal-fit/entitlements/check?sessionId=xxx
```

响应：

```json
{
  "unlocked": true,
  "productCode": "goal_fit_full_report"
}
```

完整报告页未来必须通过这个接口判断是否可以展示完整报告。`localStorage` 只能作为前端体验缓存，不能作为最终权限依据。

### 7.5 关闭订单，可选

```http
POST /api/goal-fit/orders/close
```

用途：

- 用户取消。
- 订单超时。
- 重新生成二维码前关闭旧订单。

## 8. 前端状态设计

当前 Stage 8B 的 `GoalFitUnlockPage` 状态：

```text
确认解锁完整报告
↓
前端直接 mark paid
↓
写入 unlock store
↓
进入完整报告
```

真实支付后改为：

```text
确认解锁完整报告
↓
调用 /api/goal-fit/orders/create
↓
展示二维码
↓
轮询 /api/goal-fit/orders/status
↓
status=paid 后显示完整报告已解锁
↓
进入完整报告
```

页面状态建议：

| 状态 | 用户可见文案 |
| --- | --- |
| idle | 解锁完整目标适配报告 |
| creating_order | 正在生成解锁二维码 |
| qr_ready | 请使用微信扫码完成解锁 |
| polling | 等待支付确认 |
| paid | 完整报告已解锁 |
| expired | 二维码已过期，请重新生成 |
| failed | 订单状态异常，请稍后重试 |

说明：

- Stage 8B 禁止出现“微信支付”相关文案，因为还没有接真实支付。
- Stage 8D/8E 之后接入真实支付时，可以出现“微信扫码支付”。
- 用户可见文案不要出现“模拟支付”或“测试支付”。

## 9. 权限与解锁设计

当前完整报告页依赖前端 `localStorage` 判断 unlock。真实支付上线后，这不够安全。

目标改造：

1. 完整报告页加载时，请求：

```http
GET /api/goal-fit/entitlements/check?sessionId=xxx
```

2. 如果 `unlocked=true`：

展示完整报告。

3. 如果 `unlocked=false`：

展示锁定页，并引导到解锁确认页。

4. 如果接口异常：

展示“暂时无法确认解锁状态，请稍后重试”。

5. 如果 `localStorage` 显示已解锁，但服务端 entitlement 不存在：

以服务端为准。

6. `localStorage` 只用于减少前端等待，不做最终权限判断。

### 9.1 订单与 session 绑定规则

1. 一个 session 可以有多个订单记录。
2. 但同一时间只应有一个 active pending order。
3. 一个 session 只需要一个 active entitlement。
4. paid 订单创建 entitlement 后，不允许重复解锁。
5. 重复支付风险需要通过 `out_trade_no` 唯一性和 pending 订单复用降低。
6. 前端不要自行生成金额。
7. 后端不要相信前端价格和商品名称。

## 10. 异常场景

### 10.1 用户扫码后关闭页面

处理：

再次进入解锁页时查询订单状态。如果订单已经 paid，并且 entitlement 已 active，则显示已解锁。

### 10.2 微信回调已成功，但前端轮询还没拿到

处理：

继续轮询。刷新页面后重新查询订单状态。

### 10.3 前端显示 pending，但微信已经支付

处理：

第一版先依赖回调 + 轮询。后续可新增主动查单接口作为补偿。

### 10.4 回调重复通知

处理：

幂等更新。订单已 paid 时，直接返回成功。

### 10.5 金额不一致

处理：

不创建 entitlement，记录异常，保留回调记录。

### 10.6 session 不存在

处理：

不创建订单，引导用户重新测试。

### 10.7 订单过期

处理：

允许重新生成订单。旧订单状态更新为 expired 或 closed。

### 10.8 用户重复点击确认解锁

处理：

复用未过期 pending 订单，不重复创建。

### 10.9 用户换设备

处理：

如果没有账户体系，只能通过 `sessionId` 链接恢复。未来可以增加手机号、邮箱或微信 openid 绑定。

## 11. 安全要求

1. 微信支付商户密钥不能进入前端。
2. API v3 密钥不能进入前端。
3. 商户私钥不能进入前端。
4. 前端不能直接调用微信支付下单接口。
5. 所有金额必须服务端确定。
6. 所有支付成功必须以后端回调确认为准。
7. notify 回调必须验签和解密。
8. 订单更新必须幂等。
9. entitlement 创建必须幂等。
10. 日志里不能打印私钥、API v3 key、完整支付密文。
11. 回调接口必须使用 HTTPS。
12. 数据库中要保留必要审计信息。

## 12. 环境变量

后续需要的环境变量建议如下：

```text
WECHAT_PAY_APP_ID
WECHAT_PAY_MCH_ID
WECHAT_PAY_MCH_SERIAL_NO
WECHAT_PAY_PRIVATE_KEY
WECHAT_PAY_API_V3_KEY
WECHAT_PAY_NOTIFY_URL
WECHAT_PAY_NATIVE_DESCRIPTION
WECHAT_PAY_PRODUCT_CODE
GOAL_FIT_FULL_REPORT_AMOUNT
DATABASE_URL
PUBLIC_BASE_URL
```

说明：

1. 所有 `WECHAT_PAY_*` 只能在服务端环境变量中使用。
2. 不得以 `VITE_` 前缀暴露到前端。
3. Vercel 中应配置为 Serverless Function 可读取的环境变量。
4. `WECHAT_PAY_PRIVATE_KEY` 如果使用多行 PEM，需要在环境变量中明确换行处理方式，例如使用转义换行并在服务端恢复，或使用 Vercel 支持的多行 Secret 配置方式。

## 13. 分阶段实施计划

### Stage 8D：支付后端骨架 + Mock Payment Provider

目标：

1. 新增 API 路由结构。
2. 新增数据库表迁移或 schema 文档。
3. 将 `GoalFitUnlockPage` 从纯前端 localStorage 改成调用后端 mock order。
4. Mock paid 后写入服务端 entitlement。
5. 完整报告页开始通过 entitlement API 判断权限。

边界：

不接微信支付。

### Stage 8E：微信支付 Native 真实下单和二维码展示

目标：

1. 接入微信支付 Native 下单。
2. 生成真实 `code_url`。
3. 前端展示二维码。
4. 订单状态 pending / expired / failed 正常。

### Stage 8F：支付回调 notify + 验签 + 解锁

目标：

1. 接入支付回调。
2. 验签、解密、更新订单。
3. 创建 entitlement。
4. 前端轮询后自动显示解锁成功。

### Stage 8G：线上联调与异常处理

目标：

1. 真实小额支付测试。
2. 重复回调测试。
3. 支付后刷新测试。
4. 订单过期测试。
5. 换浏览器 / 换设备恢复策略测试。

## 14. 结论

1. 固定微信收款码不适合作为本产品正式支付方式。
2. 第一版应采用微信支付 Native 二维码支付。
3. 必须增加后端和数据库。
4. 完整报告权限必须从 `localStorage` 迁移到服务端 entitlement。
5. Stage 8B 当前完成的是支付前端壳子，不是支付系统。
6. 下一阶段不应直接接微信支付，而应先做 Stage 8D：后端订单与权限 Mock 化。
7. 只有 Mock 订单和服务端 entitlement 跑通后，才接真实微信支付。
